# 自动再平衡 (Auto Rebalance) 说明

## 1. 概述

自动再平衡是当持仓市值变化后，根据当前市值占比自动调整持仓梯队归属的算法。触发时机：
- 每次加仓/减仓/清仓操作后
- 价格刷新后
- 数据导入后

## 2. 梯队配置

| 梯队 | 名称 | 目标比例 | 比例范围 | 主位上限 | 缓冲位上限 |
|------|------|-----------|----------|----------|------------|
| T1 | 第一梯队 | 30% | ≥25% | 1 | 0 |
| T2 | 第二梯队 | 20% | ≥15% 且 <25% | 2 | 1 |
| T3 | 第三梯队 | 10% | ≥5% 且 <15% | 3 | 3 |

**边界规则**：25%属于T1，15%属于T2，5%属于T3。

> **术语说明**：梯队数字越小级别越高，T1 > T2 > T3。
> - T1→T2：级别下降（数字变大）
> - T2→T1：级别上升（数字变小）
> 
> 为避免混淆，本文用"进入"或"移到"来描述，例如"T3移到T2"，而不是"T3升级到T2"。

## 3. 核心逻辑

### 3.1 判定目标梯队

根据持仓市值占总资产的比例，计算目标梯队：

```
getTargetTier(percent):
  if percent >= 25 → T1
  if percent >= 15 → T2
  if percent >= 5 → T3
  else → 最后一个梯队
```

### 3.2 简化算法（一轮完成）

**2025年优化**：从循环100次改为一轮完成。

```
autoRebalance(positions, total, cashCurrency, exchangeRates):
  1. 计算每个持仓的结算市值和占比
  2. 根据占比计算目标梯队 (getTargetTier)
  3. 按结算市值从高到低排序
  4. 第一步：分配主位
     - 遍历排序后的持仓
     - 如果主位有空位，进入主位
     - 否则保持原 inBuffer 不变
  5. 第二步：计算缓冲位可用数量
     - 缓冲位可用数量 = 所有更高权重梯队的主位空位之和
     - 例如 T3 缓冲位可用 = T1主位空位 + T2主位空位
  6. 第三步：分配缓冲位（跳过已在主位的股票）
  7. 第四步：缓冲位 promote 到主位
     - 同梯队缓冲位按市值排序，优先晋升市值大的
  8. 返回结果
```

### 3.3 缓冲位规则

**缓冲位可用数量 = 所有更高权重梯队的主位空位之和**

举例（T1=1, T2=2, T3=2, T4=2）：
- T1 主位 0/1 空1 → T2 缓冲位可用1个
- T2 主位 0/2 空2 → T3 缓冲位可用2个
- T3 主位 1/2 空1 → T4 缓冲位可用1个
- T4 主位 2/2 满 → 缓冲位可用0个

### 3.4 特殊情况处理

- **市值不足最低梯队**：如果占比小于所有梯队下限，归入最后一个梯队
- **主位满了保持原位**：如果目标梯队主位满了，该持仓保持原位置不变（不进入缓冲位）

## 4. 触发条件

```javascript
// 检查是否需要 rebalance
needsRebalance = positions.some(p => 
  p.tier !== getTargetTier(percent)
);

// 检查是否需要 promote 缓冲位
hasBufferPromotion = positions.some(p =>
  p.inBuffer && 同梯队主位数 < limit
);

// 如果都没变化，直接返回原数组（避免无意义更新）
```

## 5. 注意事项

1. **一轮完成**：算法只需执行一次，不会死循环
2. **结算货币排序**：按结算货币计算市值后排序，确保多币种持仓正确排序
3. **触发时机**：加仓/减仓/清仓/价格刷新/导入数据后自动触发
4. **深拷贝**：输入数组会被深拷贝，避免修改原对象
5. **总资产计算**：`total = 股票总市值 + 现金`（均转换为结算货币）

## 6. 建仓检查逻辑

### 6.1 检查顺序

1. **股票数是否已满**：股票数 >= 所有梯队主位之和
2. **是否超过可买梯队上限**：目标梯队 > maxTier 时拒绝
3. **梯队主位是否已满**：该梯队持仓数 >= 主位上限
4. **单笔是否超过上限**：单笔占比 > 上限占比时提示

### 6.2 maxTier 限制

`topTierAllowBuy` 配置限制最高可买梯队，但 targetTier 按实际占比计算：
- 如果 calculatedTier > maxTier，拒绝建仓
- 提示："超过可买梯队上限，当前只能买第X梯队"

## 5. 算法流程图

```
开始
  ↓
计算每个持仓的市值占比
  ↓
根据占比计算目标梯队 (getTargetTier)
  ↓
按市值从高到低排序
  ↓
初始化主位槽位和缓冲位槽位
  ↓
遍历每个持仓:
  ├─ 已在目标梯队?
  │   └─ 是: 保持当前位置
  │       └─ 原在缓冲位但主位有空? → 移入主位
  │
  ├─ 需要升级/降级?
  │   ├─ 目标梯队主位有空? → 进入主位
  │   └─ 主位已满 → 检查缓冲位
  │       ├─ 缓冲位有空? → 进入缓冲位
  │       └─ T2→T3 特殊检查 → 通过? 进入缓冲位 : 保持原位
  ↓
填充阶段: 缓冲位高市值补主位空缺
  ↓
检查是否全部稳定 (所有持仓已在目标梯队)
  ├─ 是 → 结束
  └─ 否 → 重复上述流程 (最多100次)
  ↓
返回结果
```

## 6. 注意事项

1. **迭代次数**：最多迭代 100 次，防止死循环
2. **深拷贝**：输入数组会被深拷贝，避免修改原对象
3. **总资产计算**：`total = 股票总市值 + 现金`
4. **触发时机**：加仓/减仓/清仓/价格刷新/导入数据后自动触发

### 测试真实数据场景

**场景**: 用户加仓NVDA

**原始数据**:
```json
{"positions":[{"symbol":"GOOG","value":19890.36,"tier":2,"inBuffer":false},{"symbol":"AAPL","value":20056.96,"tier":2,"inBuffer":false},{"symbol":"NVDA","value":9997.39,"tier":3,"inBuffer":false}],"cash":50055.29}
```

**加仓后数据 (传给autoRebalance)**:
- GOOG: 19890.36, tier=2, inBuffer=false
- AAPL: 20056.96, tier=2, inBuffer=false
- NVDA: 19994.78, **tier保持3**, inBuffer=false
- total: 100000

**关键**: 加仓后 tier 必须保持原来的值（不是自动计算的新值），让 autoRebalance 判断是否晋级。

**预期结果**: NVDA 晋级到 T2，应进缓冲位 (inBuffer=true)

```javascript
addRebalanceTest('ADD_NVDA', [
```

- 加仓后只更新 shares、avgCost、price、value
