# 自动再平衡 (Auto Rebalance) 说明

## 1. 概述

自动再平衡是当持仓市值变化后，根据当前市值占比自动调整持仓梯队归属的算法。触发时机：
- 每次加仓/减仓/清仓操作后
- 价格刷新后
- 数据导入后

## 2. 梯队配置

| 梯队 | 名称 | 目标比例 | 比例范围 | 主位上限 | 缓冲位上限 |
|------|------|-----------|----------|----------|------------|
| T1 | 第一梯队 | 30% | ≥25% | 1 | 0 |
| T2 | 第二梯队 | 20% | ≥15% 且 <25% | 2 | 1 |
| T3 | 第三梯队 | 10% | ≥5% 且 <15% | 3 | 3 |

**边界规则**：25%属于T1，15%属于T2，5%属于T3。

> **术语说明**：梯队数字越小级别越高，T1 > T2 > T3。
> - T1→T2：级别下降（数字变大）
> - T2→T1：级别上升（数字变小）
> 
> 为避免混淆，本文用"进入"或"移到"来描述，例如"T3移到T2"，而不是"T3升级到T2"。

## 3. 核心逻辑

### 3.1 判定目标梯队

根据持仓市值占总资产的比例，计算目标梯队：

```
getTargetTier(percent):
  if percent >= 25 → T1
  if percent >= 15 → T2
  else → T3
```

### 3.2 分配优先级

高市值持仓优先占据主位，低市值持仓进入缓冲位。

### 3.3 分配规则

1. **已在本梯队**：保持当前位（主位/缓冲位）
2. **需要移到更高梯队**（T3→T2 或 T2→T1）：
   - 优先进入目标梯队主位
   - 若主位满，进入目标梯队缓冲位
3. **需要移到更低梯队**（T1→T2 或 T2→T3）：
   - 优先进入目标梯队主位
   - 若主位满，进入目标梯队缓冲位
4. **缓冲位补充**：循环结束后，将缓冲位高市值持仓补入主位空缺
5. **目标梯队满了**：如果目标梯队主位和缓冲位都满了，尝试放到次优梯队（target + 1）

### 3.4 特殊约束

- T3->T2 降级时，若 T1+T2 主位数 ≥3，则不能进入 T2 缓冲位（因 T1+T2 最多3个）

### 3.5 目标梯队满了的处理

当持仓目标梯队的主位和缓冲位都满了时，尝试放到次优梯队（target + 1）：

- **T1 满了** → 尝试放 T2
- **T2 满了** → 尝试放 T3

例如：AAPL 占比 33%，目标 T1，但 T1 已满 → 放入 T2

## 4. 测试用例

### 用例1：全部降级到T3
**输入**：
| 股票 | 市值 | 原梯队 |
|------|------|--------|
| A | 2000 | T2 |
| B | 1800 | T2 |
| C | 1600 | T2(缓冲) |

总资产: 30000

**计算**：
- A: 2000/30000 = 6.67% → T3
- B: 1800/30000 = 6% → T3
- C: 1600/30000 = 5.33% → T3

**输出**：
| 股票 | 新梯队 | 位置 |
|------|--------|------|
| A | T3 | 主位 |
| B | T3 | 主位 |
| C | T3 | 主位 |

## 5. 算法流程图

```
开始
  ↓
计算每个持仓的市值占比
  ↓
根据占比计算目标梯队 (getTargetTier)
  ↓
按市值从高到低排序
  ↓
初始化主位槽位和缓冲位槽位
  ↓
遍历每个持仓:
  ├─ 已在目标梯队?
  │   └─ 是: 保持当前位置
  │       └─ 原在缓冲位但主位有空? → 移入主位
  │
  ├─ 需要升级/降级?
  │   ├─ 目标梯队主位有空? → 进入主位
  │   └─ 主位已满 → 检查缓冲位
  │       ├─ 缓冲位有空? → 进入缓冲位
  │       └─ T2→T3 特殊检查 → 通过? 进入缓冲位 : 保持原位
  ↓
填充阶段: 缓冲位高市值补主位空缺
  ↓
检查是否全部稳定 (所有持仓已在目标梯队)
  ├─ 是 → 结束
  └─ 否 → 重复上述流程 (最多100次)
  ↓
返回结果
```

## 6. 注意事项

1. **迭代次数**：最多迭代 100 次，防止死循环
2. **深拷贝**：输入数组会被深拷贝，避免修改原对象
3. **总资产计算**：`total = 股票总市值 + 现金`
4. **触发时机**：加仓/减仓/清仓/价格刷新/导入数据后自动触发

## 7. 用例设计指南

### 测试真实数据场景

**场景**: 用户加仓NVDA

**原始数据**:
```json
{"positions":[{"symbol":"GOOG","value":19890.36,"tier":2,"inBuffer":false},{"symbol":"AAPL","value":20056.96,"tier":2,"inBuffer":false},{"symbol":"NVDA","value":9997.39,"tier":3,"inBuffer":false}],"cash":50055.29}
```

**加仓后数据 (传给autoRebalance)**:
- GOOG: 19890.36, tier=2, inBuffer=false
- AAPL: 20056.96, tier=2, inBuffer=false
- NVDA: 19994.78, **tier保持3**, inBuffer=false
- total: 100000

**关键**: 加仓后 tier 必须保持原来的值（不是自动计算的新值），让 autoRebalance 判断是否晋级。

**预期结果**: NVDA 晋级到 T2，应进缓冲位 (inBuffer=true)

```javascript
addRebalanceTest('ADD_NVDA', [
  { symbol: 'GOOG', value: 19890.36, tier: 2, inBuffer: false },
  { symbol: 'AAPL', value: 20056.96, tier: 2, inBuffer: false },
  { symbol: 'NVDA', value: 19994.78, tier: 3, inBuffer: false }
], 100000, [
  { symbol: 'GOOG', tier: 2, inBuffer: false },
  { symbol: 'AAPL', tier: 2, inBuffer: false },
  { symbol: 'NVDA', tier: 2, inBuffer: true }
]);
```

### 前端实现注意事项

在 addPosition/adjustPosition 函数中：
- 加仓后只更新 shares、avgCost、price、value
- **不要**在这里计算新的 tier 和 inBuffer
- 保持原来的 tier 和 inBuffer，让 autoRebalance 重新判断
